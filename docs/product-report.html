Women Leaders in Digital Health Database

Product Report — June 2026

This report documents the technical approach, design decisions, and deliverables for the Women Leaders in Digital Health Database, developed for Transform Health. It responds to each of the three workstreams defined in the contract.

A. Technical & Visualisation Approach Recommendations

Assessment of Platforms and Tools

Several platforms were evaluated for building and hosting the database. The assessment considered cost, licensing, maintenance overhead, scalability, and the ability to deliver an interactive, searchable directory without ongoing licensing fees for Transform Health.

PlatformAssessmentVerdict
Google Sheets + Apps Script Used in the initial prototype. Functional for small datasets but lacks proper querying (reads all rows to filter), has a 6-minute execution limit, requires manual redeploy on every code change, and cannot handle photo uploads. Rejected
Tableau / Power BI Strong visualisation tools but designed for dashboards and reporting, not for interactive profile browsing with CRUD operations. Require per-user licensing (Tableau Creator ~$75/user/month; Power BI Pro ~$14/user/month). Not suitable as a public-facing interactive database. Rejected
Firebase Evaluated as a potential backend. Firebase Storage was partially configured but the broader platform (Firestore, Auth) introduced vendor lock-in and its document model is less suited for relational leader data. Deprioritised in favour of a unified platform. Rejected
Supabase (chosen) Open-source Firebase alternative providing PostgreSQL database, file storage, authentication, and edge functions in a single platform. Generous free tier, transparent pricing, no per-user licensing. PostgreSQL offers proper relational queries, indexing, and full-text search. Selected
React + Vite (frontend) Modern component-based framework. Vite provides fast builds and hot-module reloading during development. Tailwind CSS enables consistent styling with the Transform Health brand palette. No licensing cost. Selected

Recommended Architecture

Stack: React 18 + Vite (frontend) → Supabase PostgreSQL (database) + Supabase Storage (photos) + Supabase Auth (admin login) + Supabase Edge Functions (email sending). Hosted on GitHub Pages via CI/CD.

The architecture uses a static frontend + Backend-as-a-Service (BaaS) model:

  • The frontend (React) is built into static files and deployed to GitHub Pages — no server to maintain, no hosting cost.
  • Supabase handles all backend concerns: data queries, file storage, authentication, and server-side email sending.
  • The frontend calls Supabase's auto-generated REST API directly using the anon (public) key, with Row Level Security enforcing data access rules on the database side.

Maintenance Considerations

  • ✓ No runtime server: The frontend is a static site — no patching, no uptime monitoring, no SSH access needed.
  • ✓ CI/CD pipeline: Pushing to the main branch triggers an automatic build and deploy via GitHub Actions. No manual deployment steps.
  • ✓ Database backups: Supabase provides point-in-time recovery on paid plans. On the free tier, take periodic SQL exports via the Supabase dashboard.
  • ✓ Photo storage: Supabase Storage handles file uploads with built-in CDN. No image processing server required.
  • ○ Dependency updates: npm dependencies (React, Tailwind, etc.) should be updated periodically for security patches — standard frontend maintenance overhead.

Accessibility

The interface follows basic accessibility best practices:

  • Colour contrast: All text/background combinations meet WCAG AA contrast ratios, verified against the Transform Health brand palette.
  • Focus indicators: Visible :focus-visible outlines on all interactive elements with custom brand-colour styling.
  • Reduced motion: A @media (prefers-reduced-motion: reduce) rule disables animations for users who request it.
  • Skip link: A "Skip to main content" link is the first focusable element on every page.
  • Responsive: All layouts adapt to viewport width, tested from 320px to desktop.
  • Semantic HTML: ARIA attributes used on interactive controls (e.g., aria-expanded on toggles).

Scalability

The current architecture supports hundreds of leaders on the free tier. Supabase PostgreSQL handles indexing on filtered columns (country, expertise, status) and supports full-text search. As the dataset grows, the database can be indexed further without application changes. If traffic exceeds free-tier limits, Supabase Pro ($25/month) increases the row limit from 500 MB to 8 GB database size and raises bandwidth allowances.

Embedding Within the Transform Health Website

The database includes a "Chrome toggle" feature (an eye icon at the bottom-right of every page) that hides the site header, navigation bar, and footer. This allows the database pages to be embedded in an iframe or directly integrated into the Transform Health website without visual duplication of navigation elements.

All brand assets (logo, footer icon) are hotlinked from the Transform Health WordPress site (transformhealthcoalition.org), ensuring visual consistency with the parent site. If deeper integration is needed, the React application can be served as a subdirectory or subdomain of the main Transform Health site, sharing the same header/footer components.

Non-Technical Content Management

The Admin Console is a dedicated interface for the Transform Health team to manage the database without any technical skills:

  • Review, approve, or reject pending profile submissions
  • Search, filter, and browse the full database
  • View LinkedIn click analytics for each profile
  • Process self-service update and deletion requests
  • Manage test results from the QA testing team

Additionally, leaders can manage their own profiles entirely without admin intervention via the self-service magic link system. They find their profile, request an update or deletion, and receive a secure single-use link by email — no admin approval needed.

B. Design and Development of Interactive Database

Filters and Search

The database supports multi-dimensional exploration through:

  • Dropdown filters: Continent, country, and area of expertise — each populated dynamically from the database.
  • Free-text search: Searches across name, role, organisation, bio, expertise, and country fields.
  • Sort options: Latest (by submission date), A–Z, or by expertise area.
  • Analytics page: An interactive world map with clickable regions. Selecting a region filters the leader cards below the map. A specialisation bar chart allows simultaneous filtering by region AND expertise — clicking a bar highlights the corresponding region on the map.

Profile Presentation

Leaders are presented in a card grid layout with consistent visual structure:

  • Each card shows: photo, name, role, organisation, location, and expertise tags.
  • Clicking a card opens a 1200px modal with the full profile: bio, years of experience, countries of operation, notable achievements (with links), and LinkedIn button.
  • The email address is never shown publicly — it is visible only in the Admin Console.
  • Pagination uses a "Load more" pattern (not paginated pages) for intuitive browsing.

This split between card-level scanning and modal-level depth satisfies both browsing modes:

  • High-level scanning: Quickly scroll through cards, filter by geography or expertise.
  • Deeper exploration: Click into any card to see the complete profile, bio, and achievements.

Desktop and Mobile Optimisation

  • Responsive layout with breakpoints at mobile-first design. Cards reflow from multi-column to single-column on smaller screens.
  • Filter controls adapt: on desktop they appear inline; on mobile they stack into full-width rows with clear touch targets.
  • The modal uses a bottom-sheet pattern on mobile (slides up from the bottom) and a centered overlay on desktop.
  • All interactive elements (buttons, links, dropdowns) are sized for touch interaction on mobile.

User Roles and Journeys

Public Visitor (read-only)

Search, filter, and browse the directory. Click into profiles. No login required.

Leader (self-submit)

Complete a 5-step submission form: Branch (self/nominate) → Consent → Basic Info (name, email, photo, country) → Profile Details (years, expertise, geo scope, countries, bio) → Links (LinkedIn, achievements). Submitted profiles enter a pending queue for admin review.

Leader (manage existing profile)

Click "Manage your profile" → identify via name and email → choose Update or Delete → receive a secure magic link by email → edit fields directly or confirm removal. No admin required.

Nominator

Submit a profile on behalf of someone else by providing their name and LinkedIn URL. The nominee is notified via admin outreach.

Admin

Full access to the Admin Console: review pending submissions, approve/reject profiles, manage update and delete requests, browse all entries with LinkedIn click analytics, and run QA test result dashboards.

Data Flow

  1. Leader submits profile → stored in Supabase leaders table (status: pending) + photo uploaded to Storage
  2. Admin reviews and approves (status: live) or rejects (status: rejected)
  3. Live profiles appear in the public Directory and Analytics pages
  4. Leaders manage their profiles via self-service magic link — changes saved directly to the database
  5. Admin receives notification emails when leaders update or delete their profiles

C. Handover and Documentation

How the Database Can Be Updated and Maintained

Adding new leaders

Leaders submit their own profiles via the public "Submit your profile" form. New submissions appear in the Admin Console under Profile Requests → New, where an admin can approve (publish) or reject them.

Updating existing profiles

Two paths:

  • Self-service: Leaders use "Manage your profile" → receive a magic link by email → edit their own profile directly. No admin intervention needed.
  • Admin: Admin can manually edit or delete any record from the All Entries tab in the Admin Console.

Removing a profile

Leaders can self-serve via the "Remove profile" option in the manage flow. Admin can also delete records directly from All Entries.

Magic link email system

The email system uses a Supabase Edge Function that proxies through a Google Apps Script Web App:

requestManage() → supabase.functions.invoke("send-email", { to, subject, html })
                  → fetch(APPS_SCRIPT_URL) → Google Apps Script → SMTP send

The APPS_SCRIPT_URL, GOOGLE_SMTP_USER, and GOOGLE_SMTP_PASS secrets are configured in the Supabase Edge Function settings. The Apps Script handles SMTP delivery via Google Workspace.

Limitations and Future Expansion

Current limitations

  • Admin auth gate: Temporarily bypassed for testing. Code is ready — a one-line change re-enables it. An admin user must be created in Supabase Auth first.
  • Email delivery: Currently uses Google Apps Script as an SMTP relay. For higher volume, migrating to SendGrid or Resend would be more reliable.
  • Analytics: No page-view or user-behaviour analytics are configured yet (e.g., GA4, Plausible).
  • Country-level map drilldown: The Analytics map currently supports region-level filtering only (continent), not individual countries.
  • Photo moderation: Photos are approved when the admin approves the profile. There is no separate photo moderation workflow.

Future expansion considerations

  • ORCID integration: An open researcher identity standard with a real API. Could be offered as an optional verification method for leaders with academic backgrounds.
  • Open Badges / Verifiable Credentials: Emerging W3C standard. Leaders could carry a "Transform Health Verified" credential in a digital wallet.
  • Supabase Auth magic links: The current hand-rolled magic link token (btoa(JSON.stringify({ leaderId, mode }))) could be replaced with Supabase's built-in signInWithOtp() for production-grade, one-time-use authentication.
  • Custom domain: Moving from github.io to a custom domain like database.transformhealthcoalition.org would improve branding and SEO.
  • Vercel deployment: If GitHub Pages limits become restrictive, Vercel offers simpler environment variable management and built-in analytics at no cost.
  • Export functionality: Adding CSV/PDF export of search results or profile data for research purposes.

Identity strategy — beyond LinkedIn

Current approach: LinkedIn is treated as a vanity identifier and hyperlink only. Leaders fill in their own structured data through a consent-backed form. This is the correct approach — LinkedIn's API has been progressively locked down since 2015, scraping is explicitly prohibited in their ToS, and treating it as anything more than a link is a liability.

Future options: ORCID (open researcher identity, 20M+ users), Open Badges / Verifiable Credentials (W3C standard), and email-as-identity via Supabase Auth OTP — all platform-independent and LinkedIn cannot block them.
